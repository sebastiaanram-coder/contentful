Contentful, restructured.
A future-proof content model for Helloprint, Allgifts, Drukwerkmax, Brandon and WLS.

Today, one Contentful space carries 400,000+ entries across products, pages and static content for fourteen markets and five brands — with brand identity smuggled in as a locale suffix. This proposal redesigns the model around three independent axes — language, brand, market — in a multi-space architecture that scales without the combinatorial cost.

01 · Executive summary

The locale axis is doing three jobs at once. That's the root cause of almost every Contentful pain we feel.

Helloprint's Contentful encodes three independent concepts — language, country and brand — into a single string (the locale code). The result is a combinatorial sprawl, native Contentful features (tags, taxonomy, environments, federation) are unused, and every new brand or market becomes a code-and-content migration. Untangling these axes is the single largest unlock available to us.

The one-line proposal

Split Contentful into four purpose-built spaces (Catalog, Web, Editorial, System), move brand and market off the locale axis onto Contentful's native Tag / Taxonomy system, and let locales be what they were meant to be — W3C language tags. Everything else — environments, governance, AI Actions, content blocks, the delivery layer — falls out of that decision.

03 · Locale & brand-fallback

How locales actually resolve today

There are two parallel systems in play. Contentful's native fallbackCode resolves missing translations. On top of that, the storefront uses a custom three-part resolver (PS_LOCALE_LANGUAGE, PS_LOCALE_COUNTRY, PS_LOCALE_CONTENTFUL) to pick which locale code to query in the first place.

The structural problem in one sentence

Half of those codes are W3C BCP-47 (language × country). The other half encode a brand — a totally different concept — into the same field. Adding a sixth brand requires creating one new locale per language, updating the Contentful worker, the Algolia worker, the c/shops resolver, and the PCM locale mapping. Five edits to add a single dimension.

05 · Pain points & risks

What this is costing us

Below are the ten observable consequences of the current model, grouped by who feels them.

For engineering

Combinatorial locale sprawl

Every new brand × language combination needs a new custom locale code (nl-AGBE, de-AGDE, …). Adding the sixth brand to all six languages means six new locales, six worker config updates, and a PCM mapping change — before any content moves.

For engineering

Two parallel fallback systems

Presta resolves brand & country in code (four-tier priority); Contentful resolves language fallbacks per-field. Bugs hide between them. A daily cron caches the Contentful chain into ps_locale_fallbacks — an explicit acknowledgement that the systems don't talk live.

For editors

No clear ownership of content domains

Products, page content, UI strings, FAQs, icons and theme translations all share one space. Search is noisy, draft/publish governance is one-size-fits-all, and editorial blog work needs a Contentful permission scoped to a content type, not a space.

For editors

No proper preview / staging

Blog uses an environment called blog-sandbox as its live source. There's no convention for time-boxed feature environments, no Merge App workflow, no preview API for editors to validate before publish.

For scale

400k entries in a single space

Sampling shows the count is dominated by pcmOptionsAttribute and CombbinedOptionsAttribute — configurator option values that arguably don't belong in a content CMS at all. They drive entry counts, API limits, and query latency.

For scale

Translation chain is fragile

UI strings travel from .tpl → RabbitMQ → Contentful (themeTemplate) → _commondata GitLab → ps_contentful_translations MySQL → runtime. Five stages, multiple documented drift bugs (“old json remains in git”, “republish to force resync”).

For brand

Brand identity hidden in locale code

There is no first-class concept of “brand” in Contentful today. A query for “all Drukwerkmax content” means string-matching *-DWM locales. Cross-brand reuse needs editorial copy-paste, not entry-level branching.

For brand

Non-standard locale codes are API-only

Per Confluence, codes outside Contentful's built-in list — es-custom, nl-AGBE, en-WLS — have to be created via the CMA. The UI dropdown doesn't list them. Editors can't self-serve, so brand-locale changes are always engineering tickets.

For data

PCM keeps its own mapping of CF locales

Per the “Creating a Custom Contentful Locale” runbook: “PCM maintains their own mapping of Contentful locales, so remember to let them know if you add a new locale.” Two systems hold a view of the same axis, kept in sync by a Slack message rather than an API contract.

For governance

Naming & type quality has drifted

PascalCase (CombbinedOptionsAttribute) sits next to camelCase (pcmOptionsAttribute). Cryptic IDs (homeBlockTypeBJ, homeBlockTypeBk) sit next to descriptive ones. Typos are baked into IDs and can't be renamed in place.

06 · Design principles

Five rules the new model will obey

Greenfield doesn't mean “anything goes”. The proposal below honours these five constraints. If a future tweak breaks one, push back.

Rule 1

One concept, one axis

Language, brand and market are three independent concepts. Each gets its own field, its own resolution rule, its own governance. The locale code carries language only.

Rule 2

Use Contentful natively before extending it

Where Contentful ships a feature — Tags, Taxonomy Concept Schemes, environments, fallbackCode, AI Actions — we use it. Custom workers exist only where we must.

Rule 3

Spaces follow lifecycle, not topology

Content with the same publish cadence, same editors, same approval flow goes in the same space. Catalog data and editorial blog have nothing in common — they don't share a space.

Rule 4

Brand is a tag, not a tenant

Allgifts, Drukwerkmax, Brandon and WLS share >80% of their content with Helloprint. We model brand-specific content as overrides, not as separate-but-equal copies.

Rule 5

Make adding a brand cheap

A new brand should be one Tag, optional brand-specific overrides, and a routing rule in the delivery layer. No new locale codes, no PCM mapping changes, no content-type edits.

Rule 6

Configurator data does not belong in Contentful

~half of the 400k entries are configurator option values. They have no editorial workflow. Move them to PCM proper or a purpose-built catalog store; keep Contentful for content.

08 · Locale & brand model

One concept, one axis — three times over

The single largest change. Today's combined “locale” field becomes three orthogonal fields: locale (language only), brand (tag), market (tag). Resolution happens in the delivery layer, not in the locale code.

FIG. 4 — Three orthogonal axes replace one overloaded locale. The delivery layer composes them deterministically at read time.

How a single entry shows up across brand × market

Entry tag(s)SpecificityResolved for …
no tagsMost generalEvery brand, every market. The Helloprint default.
brand:allgiftsBrand-specificAllgifts customers in any market.
market:BEMarket-specificAny brand displayed for the Belgium market.
brand:allgifts + market:BEMost specificAllgifts customers, Belgium market — wins over both single-axis matches.
What we lose

Nothing material. Brand fallback today is “if I don't have an nl-AGBE value, use the nl-BE value”. That exact behaviour is the new model's brand-agnostic fallback — just expressed in tags instead of locale strings.

What we gain

A new brand becomes one tag. A new market becomes one tag. Editors can query “all Drukwerkmax content” with a tag filter, not a string-match. Contentful's native Tags + Taxonomy do the heavy lifting. The custom four-tier resolver in c/shops goes away.

10 · Environments & governance

Environments do what they're called

Per-space environments

  • master — production. Frontends read from this.
  • staging — pre-production. CI smoke-tests run against it.
  • preview — editors' draft preview surface (Contentful Preview API).
  • feat-<ticket> — short-lived forks for content-model migrations. Merged via Contentful Merge App.

Names mean what they say. No more “sandbox that's actually production”.

Roles & permissions

  • Catalog write — PCM service account only.
  • Web edit — market editor roles, brand-scoped via Tag-based filters.
  • Editorial — Marcomms team, own approval workflow.
  • System — engineering + localisation. Versioned via CI.

Permissions are space-level, not type-level. Domain isolation does the work.

Workflows & AI Actions

Workflows

Each space defines its own Contentful Tasks + Workflows. Web has Draft → Market review → Approve → Publish. Editorial has Draft → Editor review → Legal/comms → Publish. Catalog is API-only — no manual workflow.

AI Actions (centralised)

SEO meta drafting, alt-text suggestion, translation drafting, B2B-tone rewriting. All defined in the System space, exposed read-only to others. One source of prompts; one audit trail.

11 · Delivery layer & pipelines

One gateway, fewer custom workers

The biggest operational gain of splitting axes is that frontends stop caring about brand and country in their queries. The Content Gateway resolves them.

Pipeline today

Five custom workers, three syncs

  • Contentful Worker → GitLab _commondata
  • Algolia Worker — per-country
  • Page Index — Presta crons
  • Cache Invalidator — GCF pub/sub
  • Translation sync — RabbitMQ + MySQL
  • ps_locale_fallbacks — daily cron

Every locale change ripples through all six.

Pipeline tomorrow

One gateway, two specialised syncs

  • Content Gateway (GraphQL) — federates four spaces, resolves brand × market × locale
  • Algolia indexer — consumes the gateway, not raw spaces
  • Cache invalidation — native Contentful webhook → CF purge by tag {space}.{contentType}.{entryId}.{brand}.{market}
  • UI-string sync — compile from System space to per-locale JSON bundles at build time. RabbitMQ + MySQL retired.

Cache strategy

Tag-by-composite

Today's cache tag is just entryId. Tomorrow's is {contentType}.{entryId}.{brand}.{market}. A change to a brand-specific override only purges that brand's cache — not the entire entry across every brand and market. Same plumbing (Cloudflare purge-by-tag), better targeting.

What goes away

  • ps_locale_fallbacks MySQL table and its 06:00 cron — fallback is purely Contentful-native.
  • ps_contentful_translations table — UI strings come from compiled JSON bundles.
  • The four-tier PS_LOCALE_* resolver in c/shops — brand & market are now request-level tags, not locale codes.
  • PCM's parallel locale map — Contentful is the only source.

13 · Migration sketch

A phased move, lowest-risk first

Greenfield first — but the migration matters. Five phases, sequenced so each one ships value standalone and the next builds on it. Detailed plan to follow if approved.

FIG. 5 — Phased migration. Editorial first (lowest risk, contained team), then System (parity-testable), then Catalog (PCM writes), then Web (most cross-references), then decommission.

Phase 0 · Foundations

Create four new spaces with empty models. Define and review naming conventions. Set up environments and gateway scaffolding. No content moves.

Phase 1 · Editorial

Move blog out of blog-sandbox. Smallest team, independent lifecycle, lowest risk. Proves the model.

Phase 2 · System

Migrate themeTemplate to uiString with namespaced IDs. Compile to JSON bundles. Run new and legacy pipelines in parallel until parity.

Phase 3 · Catalog

PCM writes to Catalog space. Configurator options moved out of Contentful. Legacy product entries archived. PCM's locale map retired.

Phase 4 · Web + brand-as-tag

Migrate productPage, blocks and homepages. Replace brand-locale codes (nl-AGBE, …) with brand + market Tags. The four-tier resolver retires.

Phase 5 · Decommission

Retire the original Products space, the Marketing space, the worker queue tables and the daily fallback cron. Frontends read from the gateway only.

Coexistence period

Phases 1–4 each ship with the old pipeline still running. The gateway routes per-content-type between old and new spaces during the cutover, so editors and frontends are never broken at the same time.

14 · Next steps

What to decide first

Decision 1

Do we agree brand & market belong on Tag axes, not in the locale code? Everything downstream depends on this single call.

Decision 2

Do we accept a four-space split, or should we collapse Editorial + Web for ease of cross-references?

Decision 3

Where do configurator option values live — Contentful (today), PCM, or a dedicated catalog DB?

Decision 4

Who owns the new Content Gateway — one team end-to-end, or shared between commerce platform and content engineering?

Decision 5

Migration window — can we ship phase 1 (blog) within one quarter, or do we sequence behind another initiative?

Decision 6

Brand governance — who approves a new Brand Tag (and the brand-theme entry that follows)? Marcomms? AI/CTO office?

Suggested first action

30-minute alignment with the owners of PCM, the Contentful Worker, the Algolia Worker and the Content team. Confirm Decision 1 in that meeting; everything else can be planned afterwards. If alignment holds, a Phase-0 spike to stand up the four empty spaces is ~1 week of one engineer.
